MySQL中的两种排序方式

1.通过有序索引顺序扫描直接返回有序数据

因为索引的结构是B+树，索引中的数据是按照一定顺序进行排列的，所以在排序查询中如果能利用索引，就能避免额外的排序操作。EXPLAIN分析查询时，Extra显示为Using index。

2.Filesort排序，对返回的数据进行排序

所有不是通过索引直接返回排序结果的操作都是Filesort排序，也就是说进行了额外的排序操作。EXPLAIN分析查询时，Extra显示为Using filesort。

网络分层的真实含义是什么

2018-08-01

四个问题

  1. TCP 在进行三次握手的时候，IP 层和 MAC 层对应都有什么操作呢？

    TCP 发送每一个消息，都会带着 IP 层和 MAC 层了。因为，TCP 每发送一个消息，IP 层和 MAC 层的所有机制都要运行一遍。而你只看到 TCP 三次握手了，其实，IP 层和 MAC 层为此也忙活好久了。

  2. A 知道自己的下一个中转站是 B，那从 A 发出来的包，应该把 B 的 IP 地址放在哪里呢？B 知道自己的下一个中转站是 C，从 B 发出来的包，应该把 C 的 IP 地址放在哪里呢？如果放在 IP 协议中的目标地址，那包到了中转站，怎么知道最终的目的地址是 D 呢？

  1. 二层设备处理的通常是 MAC 层的东西。那我发送一个 HTTP 的包，是在第七层工作的，那是不是不需要经过二层设备？或者即便经过了，二层设备也不处理呢？或者换一种问法，二层设备处理的包里，有没有 HTTP 层的内容呢？

    二层设备收进去的是整个网络包。这里面 HTTP、TCP、 IP、 MAC 都有。什么叫二层设备，就是只把 MAC 头摘下来，看看到底是丢弃、转发，还是自己留着。

  2. 从你的电脑，通过 SSH 登录到公有云主机里面，都需要经历哪些过程？或者说你打开一个电商网站，都需要经历哪些过程？说得越详细越好。

网络为什么要分层？

因为复杂的程序都要分层。所谓的二层设备、三层设备，都是这些设备上跑的程序不同而已。始终想象自己是一个处理网络包的程序，如何拿到网络包、如何根据规则进行处理、如何把包发出去。

程序是如何工作的？

想象自己是一个处理网络包的程序，接收到Buffer时，调用process_layer2(buffer)函数，摘掉二层的头（MAC头），如果MAC地址和自己的相符，则说明包是发送给自己的。继续调用process_layer3(buffer)函数，此时Buffer里已经没有二层的头（MAC头）了。如果IP地址不是自己的，那就应该转发出去；如果IP地址是自己的，就是发送给自己的。摘掉三层的头（IP头），继续交给上一层处理。假设上层是基于TCP，则调用process_tcp(buffer)函数。此时Buffer中已经没有三层的头（IP头），所以需要查看四层的头（TCP头）。看这是发起还是应答，或者是一个普通的数据包。如果是发起或者应答，则接下来可能要发送一个回复包。如果是一个普通数据包，则交由上层处理（process_http(buffer)函数）。或者交由应用程序程序自己去处理，交给哪个应用程序呢？在四层的头（TCP头）里有端口号，不同的程序的监听不同的端口号。我们假设交给浏览器去处理，浏览器解析HTML，展示出页面。

此时用户点击页面上的按钮，浏览器发起一次新的HTTP请求。首先调用send_tcp(buffer)函数，buffer就是HTTP请求内容，这个函数会给buffer加一个TCP头，记录下源端口号和目标端口号（浏览器会提供）。然后调用send_layer3(buffer)函数，此时buffer中包含了HTTP的请求内容和请求头以及TCP头，在这个函数里加一个IP头，记录下源IP地址和目标IP地址。然后调用send_layer2(buffer)函数，buffer里面已经包含了HTTP的请求内容和请求头、TCP头以及IP头，在这个函数里面加一个MAC头，记录源MAC地址和目标MAC地址，此时目标MAC地址不一定能知道，如果不知道的话，需要通过ARP协议获取MAC地址。MAC地址是必须要有的。

层与层之间的关系

层层封装的关系，可以想象为套娃。只要是在网络上跑的包，都是完整的。可以有下层没上层，但是绝对不可能有上层没下层。

MySQL中能够使用索引的场景

2018-07-29

匹配全值

对索引中所有列都有指定具体值，即是对索引中的所有列都有等值匹配的条件。

rental表的索引情况，其中idx_rental_date是复合索引：

EXPLAIN SELECT * FROM rental WHERE rental_date='2005-05-25' AND inventory_id=373 AND customer_id=343;

优化器选择了复合索引idx_rental_date

匹配值的范围查询

对索引的值进行范围查找。

EXPLAIN SELECT * FROM rental WHERE customer_id>373 AND customer_id<400;

优化器选择索引范围查询，使用索引idx_fk_customer_id加速访问。注意Extra列为Using index condition，表明MySQL使用了ICP进一步优化查询。关于ICP后面会单独介绍。

匹配最左前缀

MySQL中B-Tree索引使用的首要原则——最左匹配原则

对于复合索引而言，MySQL仅仅使用索引的最左列进行查找。比如在col1+col2+col3字段上的联合索引能够被包含col1、（col1+col2)、（col1+col2+col3）的等值查询利用到，但是不能够被col2、col3、（col2+col3）的等值查询利用到。

如果查询条件中包含了索引的第一列rental_date和第三列customer_id，从key中可以看出优化器使用了复合索引idx_rental_date进行条件过滤：

EXPLAIN SELECT * FROM rental WHERE rental_date='2005-05-25' AND customer_id=343;

如果查询条件中包含了索引的第二列inventory_id和第三列customer_id，从key中可以看出优化器使用了其各自的索引进行条件过滤，但是进行了索引合并优化（type=index_merge）：

EXPLAIN SELECT * FROM rental WHERE inventory_id=373 AND customer_id=343;

仅仅对索引进行查询

当查询的列都在索引的字段中时，查询的效率更高。

EXPLAIN SELECT customer_id FROM rental;

优化器选择了查询列的索引加速访问，并且Extra为Using index意味着直接访问索引就足够取到所需要的数据，不会再通过索引回表查询。

匹配列前缀

使用只包含索引字段开头一部分进行数据查找

为customer表建立前缀索引：

查看customer表的索引情况：

EXPLAIN SELECT email FROM customer WHERE email LIKE 'MARY%';

可以看到，前缀索引idx_email已经被利用上了，Extra为Using where表明优化器还需要通过索引回表查询数据，因为索引的只是email字段的前5个字符。

如果列名是索引，那么使用column_name is null 就会使用索引（区别于Oracle）

EXPLAIN SELECT * FROM payment WHERE rental_id IS NULL;

ICP优化

MySQL5.6引入了Index Condition Pushdown（ICP）的特性，进一步优化了查询。Pushdown表示操作下放，某些情况下的条件过滤操作下放到存储引擎。

EXPLAIN SELECT * FROM rental WHERE rental_date='2005-05-25' AND customer_id>=300 AND customer_id<=400;

在5.6版本之前：

优化器首先使用复合索引idx_rental_date过滤出符合条件rental_date='2005-05-25'的记录，然后根据复合索引idx_rental_date回表获取记录，最终根据条件customer_id>=300 AND customer_id<=400过滤出最后的查询结果（在服务层完成）。

在5.6版本之后：

MySQL使用了ICP来进一步优化查询，在检索的时候，把条件customer_id>=300 AND customer_id<=400也推到存储引擎层完成过滤，这样能够降低不必要的IO访问。Extra为Using index condition就表示使用了ICP优化。
